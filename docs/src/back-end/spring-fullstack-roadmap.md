# 前端转 Spring 全栈学习路线

> 这条路线不是让你从零背完 Java 语法，而是帮你建立“能用大模型做后端项目，并且能判断结果对不对”的能力。

## 你最终要具备什么能力

学完这条路线，你应该能做到：

- 看懂一个 Spring Boot 项目的基本结构。
- 根据前端页面反推出需要哪些接口。
- 设计简单可靠的数据库表。
- 让大模型生成 Controller、Service、Mapper、DTO 等代码。
- 检查大模型代码有没有分层混乱、权限缺失、事务错误、异常处理不统一等问题。
- 用接口工具跑通流程，并能定位常见错误。
- 把前端、后端、数据库一起跑起来，完成基本部署。

你的目标不是成为“手写所有代码的人”，而是成为“能把业务功能从页面管到数据库和上线的人”。

## 学习方式

大模型时代，学习后端不要按传统路线死磕语法。更合适的方式是：

```text
业务需求
-> 接口设计
-> 数据建模
-> 让大模型生成代码
-> 审查代码
-> 接口测试
-> 前后端联调
-> 部署上线
```

每学一个知识点，都要落到一个小功能里。不要只看文章，不动手。

每天可以按这个节奏：

```text
30 分钟：学一个概念
60 分钟：做一个小功能
20 分钟：接口测试和排错
10 分钟：总结今天踩的坑
```

## 第 1 周：Java 最小基础

目标：不是学完 Java，而是能读懂 Spring Boot 代码。

要掌握：

- 类、对象、方法。
- 接口和实现类。
- 注解。
- 泛型。
- `List`、`Map`。
- 异常处理。
- Lambda 和 Stream 的常见写法。

先看这些笔记：

- [基础语法：Java vs JS](/back-end/java/02-syntax-basics.md)
- [面向对象核心](/back-end/java/03-oop-core.md)
- [接口与抽象类](/back-end/java/03a-interface-abstract.md)
- [集合框架](/back-end/java/04-collections.md)
- [Lambda 与 Stream](/back-end/java/04a-lambda-stream.md)
- [异常处理](/back-end/java/05-exception.md)
- [注解入门](/back-end/java/05a-annotation-basics.md)

本周练习：

```text
让大模型给你写一个 User 类、UserService 接口、UserServiceImpl 实现类。
你要能说清楚每个类负责什么，为什么接口和实现类要分开。
```

验收标准：

- 能看懂 `Controller -> Service -> Mapper` 的调用方向。
- 能分清类、接口、注解、泛型的基本作用。
- 能看懂常见的 `List<User>`、`Map<String, Object>`。

## 第 2 周：HTTP 和接口设计

目标：能从前端页面反推出后端接口。

要掌握：

- `GET`、`POST`、`PUT`、`PATCH`、`DELETE`。
- path 参数、query 参数、body 参数、header 参数。
- 常见状态码：`200`、`201`、`400`、`401`、`403`、`404`、`409`、`500`。
- 分页、搜索、排序接口怎么设计。

推荐先看：

- [一个请求的完整链路](/back-end/frontend-backend-guide/03-request-lifecycle)

本周练习：给“任务管理页面”设计接口。

```text
GET    /api/tasks              查询任务列表
GET    /api/tasks/{id}         查询任务详情
POST   /api/tasks              创建任务
PUT    /api/tasks/{id}         更新任务
DELETE /api/tasks/{id}         删除任务
PATCH  /api/tasks/{id}/status  修改任务状态
```

你还要写出请求和响应：

```json
{
  "title": "完成首页开发",
  "description": "完成首页 UI 和接口联调",
  "priority": "HIGH",
  "dueDate": "2026-05-20"
}
```

验收标准：

- 看到一个列表页，能说出需要列表接口、详情接口、新增接口、编辑接口、删除接口。
- 能判断参数应该放在 query、path 还是 body。
- 能给接口设计合理的错误返回。

## 第 3 周：MySQL 和数据建模

目标：能为常见功能设计表。

要掌握：

- 表、字段、主键。
- 唯一约束。
- 索引。
- 一对一、一对多、多对多。
- 软删除。
- 创建时间和更新时间。
- 事务的基本意义。

先看这些笔记：

- [数据库基本概念](/back-end/database/basics/concepts.md)
- [关系型 vs 非关系型](/back-end/database/basics/relational-vs-nosql.md)
- [SQL 基础](/back-end/database/mysql/sql-basics.md)
- [高级查询](/back-end/database/mysql/advanced.md)

本周练习：设计任务管理系统的表。

```sql
users
- id
- username
- email
- password_hash
- status
- created_at
- updated_at

tasks
- id
- user_id
- title
- description
- priority
- status
- due_date
- deleted
- created_at
- updated_at
```

验收标准：

- 能解释为什么 `email` 要唯一。
- 能解释为什么任务表要有 `user_id`。
- 能解释为什么密码不能存明文。
- 能解释什么时候需要索引。

## 第 4 周：Spring Boot CRUD

目标：完成任务模块的增删改查。

先看这些笔记：

- [Maven](/back-end/java/06-maven.md)
- [Lombok](/back-end/java/06a-lombok.md)
- [实战：Spring Boot CRUD](/back-end/java/07-spring-boot-crud.md)
- [Spring IoC 与依赖注入](/back-end/java/07a-spring-ioc-di.md)
- [Spring Boot 配置详解](/back-end/java/07b-spring-config.md)
- [Spring Data 数据库实战](/back-end/java/08-spring-data-db.md)

建议项目结构：

```text
controller
service
service/impl
mapper
entity
dto
vo
common
exception
config
```

本周练习：实现任务 CRUD。

要求：

- Controller 只接收请求和返回结果。
- Service 写业务规则。
- Mapper 只负责数据库。
- 新增任务时默认状态为 `TODO`。
- 返回统一结构 `Result<T>`。

验收标准：

- 能跑通新增、查询、编辑、删除。
- 能解释 Controller、Service、Mapper 各自负责什么。
- Controller 里没有直接写数据库操作。

## 第 5 周：参数校验和异常处理

目标：让接口从“能跑”变成“像真实项目”。

要掌握：

- `@Valid`。
- `@NotBlank`、`@NotNull`、`@Size`。
- 全局异常处理。
- 统一返回格式。
- 业务错误码。

本周练习：给任务接口加校验。

规则：

```text
title 不能为空
title 最长 100 个字符
priority 只能是 LOW、MEDIUM、HIGH
任务不存在时返回 404
重复数据时返回 409
```

验收标准：

- 参数错了不会进业务逻辑。
- 错误返回格式统一。
- 前端能直接拿 `message` 做提示。

## 第 6 周：登录和 JWT

目标：让系统知道“当前用户是谁”。

要掌握：

- 注册。
- 登录。
- 密码哈希。
- JWT。
- 登录拦截器。
- 当前用户上下文。

本周练习：完成注册和登录。

要求：

- 密码不能明文保存。
- 登录成功返回 token。
- 创建任务时自动绑定当前用户。
- 未登录访问任务接口返回 `401`。

验收标准：

- 能解释 token 放在哪里。
- 能解释后端怎么知道当前用户是谁。
- 能解释为什么不能相信前端传来的 `userId`。

## 第 7 周：权限和越权控制

目标：避免“能登录但不安全”。

要掌握：

- 普通用户和管理员。
- 接口权限。
- 数据权限。
- 越权问题。

关键例子：

```text
用户 A 不能通过修改 taskId 删除用户 B 的任务。
```

本周练习：

- 普通用户只能看自己的任务。
- 管理员可以看所有任务。
- 普通用户不能访问管理接口。
- 用户不能删除别人的任务。

验收标准：

- 所有涉及用户数据的接口都检查数据归属。
- 未登录返回 `401`。
- 无权限返回 `403`。

## 第 8 周：前后端联调

目标：把前端页面和 Spring Boot 后端接起来。

要完成：

- 登录页。
- 任务列表页。
- 新增/编辑任务表单。
- 删除确认。
- loading 和错误提示。
- token 过期处理。

建议统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

错误时：

```json
{
  "code": 40001,
  "message": "任务标题不能为空",
  "data": null
}
```

验收标准：

- 前端能完整完成登录、查询、新增、编辑、删除。
- 登录过期后能回到登录页。
- 接口错误能给出用户看得懂的提示。

## 第 9 周：博客系统

目标：练复杂一点的数据关系。

功能：

- 文章。
- 分类。
- 标签。
- 评论。
- 点赞。
- 收藏。
- 草稿和发布。

重点练：

- 一对多：用户和文章。
- 多对多：文章和标签。
- 状态流转：草稿、已发布、已下架。
- 后台管理。

验收标准：

- 能画出表关系。
- 能解释文章、分类、标签为什么这样设计。
- 能做文章列表的分页、搜索、筛选。

## 第 10 周：订单系统

目标：接触更像真实业务的后端问题。

功能：

- 商品。
- 购物车。
- 订单。
- 库存。
- 支付状态。
- 订单取消。
- 后台发货。

重点练：

- 事务。
- 库存扣减。
- 订单状态。
- 重复提交。
- 数据一致性。

验收标准：

- 下单时库存要正确减少。
- 库存不足不能下单。
- 订单状态不能乱跳。
- 关键操作要有事务。

## 第 11 周：测试和排错

目标：能判断功能是不是真的对。

要掌握：

- Apifox / Postman。
- JUnit 基础。
- 接口测试。
- 日志查看。
- 异常堆栈阅读。
- 数据库检查。

每个接口都写验收清单，比如创建任务：

```text
title 为空，应该失败
title 超长，应该失败
正常创建，应该成功
未登录创建，应该返回 401
创建后数据库有记录
返回结果不包含 password_hash 等敏感字段
```

验收标准：

- 能根据报错判断问题大概在前端、接口、业务逻辑、SQL、权限还是配置。
- 能让大模型生成测试，但自己能判断测试有没有覆盖关键场景。

## 第 12 周：Docker 部署

目标：把项目完整跑起来。

要掌握：

- 前端构建。
- 后端打包。
- 环境变量。
- Nginx。
- MySQL。
- Docker。
- Docker Compose。

最低目标：

```text
前端 build 后由 Nginx 托管
后端 Spring Boot 打成 jar 运行
MySQL 独立运行
前端请求后端 API
```

验收标准：

- 能用一个命令启动完整系统。
- 浏览器能访问前端页面。
- 前端能正常调用后端接口。
- 重启后数据还在。

## AI 协作管控模板

### 模板 1：先要设计，不要直接写代码

```text
我要实现用户管理模块。

功能：
1. 用户列表，支持分页、关键词搜索、状态筛选。
2. 新增用户，username 和 email 必须唯一。
3. 禁用用户后不能登录。
4. 管理员不能删除自己。
5. 删除用户采用软删除。

技术：
Spring Boot + MyBatis Plus + MySQL。

请先输出：
1. 数据库表设计
2. 接口列表
3. 请求响应结构
4. 关键业务规则
5. 可能的风险点

先不要写代码。
```

### 模板 2：让大模型按分层写代码

```text
请基于上面的设计实现创建用户接口。

要求：
1. Controller 只接收请求和返回结果。
2. Service 负责业务校验。
3. Mapper 只负责数据库操作。
4. username 和 email 不能为空。
5. email 重复时返回 409。
6. 所有接口返回统一 Result<T>。
7. 不要在 Controller 里直接调用 Mapper。
```

### 模板 3：让大模型自查代码

```text
请检查这段代码是否存在以下问题：
1. Controller 业务逻辑过重
2. 参数校验缺失
3. 越权风险
4. 事务问题
5. 数据库查询性能问题
6. 异常处理不统一
7. 敏感字段泄露
```

### 模板 4：让大模型生成测试用例

```text
请为这个接口生成测试用例。

要求覆盖：
1. 正常流程
2. 参数错误
3. 未登录
4. 无权限
5. 数据不存在
6. 重复提交
7. 越权访问
```

## 每周复盘问题

每周结束时，问自己：

- 我能不能解释这个功能的数据表？
- 我能不能解释每个接口的用途？
- 我能不能说出哪些地方需要权限校验？
- 我能不能用 Apifox 或 Postman 跑通完整流程？
- 我能不能看懂报错？
- 我能不能指出 AI 代码哪里不合理？

如果这些问题答不上来，不要急着往后学。

## 暂时不要优先学什么

这些不是不重要，而是可以后置：

- Spring 源码。
- JVM 调优。
- 高并发架构。
- 微服务全家桶。
- Kubernetes。
- 复杂设计模式。
- 消息队列深层原理。
- 分布式事务。

先把单体应用做好。一个能登录、能鉴权、能操作数据库、能部署的小系统，比一堆没落地的概念更有价值。
