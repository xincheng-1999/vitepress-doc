# 垃圾回收 GC

> 上一章 [JVM 内存模型](/back-end/frontend-backend-guide/18-jvm-memory-model) 讲了内存被分成栈和堆、堆里又分新生代和老年代。这一章讲一件你写前端时几乎不用操心的事：**这些对象什么时候、被谁清掉。**
>
> 你可能会想：「JS 不也有垃圾回收吗？我从来没管过啊。」没错——前端 GC 你不用管，是因为页面活几分钟就刷新，GC 卡顿几毫秒也没人在意。但后端服务 7×24 跑几个月、堆有几个 G、还在高并发下扛着上万请求，**GC 一旦卡顿就是「所有接口同时变慢几百毫秒」**。这就是后端工程师必须懂 GC 的根本原因。

这一章不教你背收集器参数（那是高级调优的事），而是让你能做到三件事：**听懂 GC 在说什么**、**看一段 GC 日志判断它健康不健康**、**知道 Full GC 频繁时该往哪查**。

---

## 19.1 为什么需要 GC：你不 `free`，谁来 `free`

在 C/C++ 里，你 `malloc` 了一块内存，就得自己记得 `free`，忘了就内存泄漏，free 早了就野指针崩溃。这是手动内存管理，心智负担极重。

Java（和 JS 一样）走的是**自动内存管理**：你只管 `new`，不管释放。JVM 有一个后台机制——**GC（Garbage Collector，垃圾回收器）**——专门负责找出「再也用不到的对象」，把它们占的内存收回来，给后续的 `new` 腾地方。

> **前端类比**：这跟 V8（Node/浏览器的 JS 引擎）的 GC 是一回事。你写 `const obj = {}` 从不手动销毁，等没人引用它了，V8 自己回收。Java 的 GC 思路完全一样，区别只在于：**JVM 的 GC 你可以观测、可以调、出问题了能看日志**——而前端的 V8 GC 基本是个黑盒。

那么问题来了：**JVM 怎么知道一个对象「再也用不到了」?**

---

## 19.2 怎么判断对象该不该回收：可达性分析

### 一个错误但直觉的方案：引用计数

最直觉的想法是「引用计数」：每个对象记一个数字，被引用一次加一，引用断开减一，数字归零就回收。Python 早期、很多脚本语言都用过它。

它有个**致命缺陷：循环引用**。看这个例子：

```java
class Node {
    Node next;   // 指向另一个 Node
}

Node a = new Node();
Node b = new Node();
a.next = b;      // a 引用了 b
b.next = a;      // b 引用了 a
a = null;        // 业务代码不再使用 a
b = null;        // 业务代码不再使用 b
```

`a` 和 `b` 这两个局部变量都置 null 了，外界谁都用不到这两个对象。但如果用引用计数：`a` 对象还被 `b.next` 引用着（计数 1），`b` 对象还被 `a.next` 引用着（计数 1）——两个计数都不为零，于是**永远回收不掉**，内存泄漏。

> **前端类比**：你在 React 里两个对象互相 `ref` 对方，组件卸载后却没断开引用——如果引擎只看引用计数，这俩就泄漏了。现代 JS 引擎也不用引用计数，用的是下面这种「可达性分析」。

### JVM 实际用的方案：可达性分析（GC Roots）

JVM 不数引用，而是**从一组「肯定还活着」的根对象出发，沿着引用链往下走，能走到的对象就是「存活」，走不到的就是「垃圾」**。这组根对象叫 **GC Roots**。

```text
GC Roots（一定活着的起点）
   │
   ├─→ 栈帧里的局部变量（正在执行的方法里的变量）
   ├─→ static 静态字段（类级别的全局引用）
   ├─→ 常量引用
   └─→ JNI（native 方法）持有的对象
        │
        ▼  沿引用链遍历（可达 = 存活）
   ┌──────────────┐        ┌──────────────┐
   │ 存活对象 A    │──ref──→│ 存活对象 B    │   ← Roots 能走到 → 保留
   └──────────────┘        └──────────────┘

   ┌──────────────┐        ┌──────────────┐
   │ 孤岛对象 X    │◄─ref──►│ 孤岛对象 Y    │   ← Roots 走不到 → 回收
   └──────────────┘        └──────────────┘
   （X、Y 互相引用，但没有任何 GC Root 能到达它们）
```

回到上面的循环引用例子：`a` 和 `b` 已经置 null，**没有任何 GC Root 能走到这两个对象**，所以可达性分析判定它们是垃圾，照样回收——循环引用的问题自然就解决了。

> **前端类比**：把 GC Roots 想成「当前页面正在用的东西」——正在渲染的组件树、挂在 `window` 上的全局变量、活跃的事件监听器。从这些出发够不到的对象，就是可以清掉的垃圾。两个互相引用但都从根够不到的对象，就是一座「孤岛」，照样被回收。

**什么时候触发回收？** 不是对象一变垃圾就立刻回收（那样太频繁、太耗 CPU），而是**等到内存不够用了才回收**：新生代放不下新对象了，触发一次新生代回收；老年代也快满了，触发一次整堆回收。所以 GC 是「攒一批一起清」，不是「随脏随扫」。

## 19.3 为什么 GC 会让接口卡顿：STW（Stop The World）

这是整章最重要的概念，也是后端必须关心 GC 的核心原因。

GC 在标记「哪些对象存活」时，有个棘手的问题：**如果业务线程一边在跑、一边在改对象之间的引用关系，GC 就像在数一群乱跑的羊，永远数不准。** 为了数准，GC 在某些关键阶段必须**把所有业务线程全部暂停**，等它处理完再放行。这个暂停就叫 **STW（Stop The World，停顿世界）**。

```text
正常情况：业务线程们各跑各的
线程1 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶
线程2 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶
线程3 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶

发生 STW：所有业务线程被冻结，只有 GC 线程在跑
线程1 ━━━━━━━┃          ┃━━━━━━━━━━━━━━━━━▶
线程2 ━━━━━━━┃  STW     ┃━━━━━━━━━━━━━━━━━▶
线程3 ━━━━━━━┃  (暂停)   ┃━━━━━━━━━━━━━━━━━▶
GC                ████████  ← 只有 GC 在工作
            └─ 这段时间，所有请求都被卡住，零响应 ─┘
```

**STW 期间发生了什么？** 此刻 `svc-canvas` 的 200 个 Tomcat 工作线程**全部冻结**，正在处理的请求全部暂停。如果这次 STW 持续 500ms，那么这 500ms 内进来的所有请求、正在处理的所有请求，全都被卡 500ms——监控上就是一根整齐的延迟尖刺，所有接口的 P99 同时抖一下。

> **前端类比**：想象浏览器主线程被一段同步代码独占 500ms——页面卡死、点击没反应、动画掉帧。STW 就是后端版的「主线程被卡住」，只不过被卡住的不是一个用户的页面，而是**这个进程上所有用户的所有请求**。这就是为什么前端 GC 卡顿无所谓（卡的是你自己一个 Tab），后端 GC 卡顿是大事（卡的是全体在线用户）。

所以后端关心 GC，本质上只关心一件事：**STW 别太长、别太频繁。** 现代收集器（如 G1）的全部努力，都是在「把垃圾收干净」和「STW 尽量短」之间找平衡。

## 19.4 Minor GC vs Full GC：频繁 vs 致命

JVM 把堆分代（见 [JVM 内存模型](/back-end/frontend-backend-guide/18-jvm-memory-model)）就是为了**分级回收**——新对象死得快，老对象活得久，分开收效率最高。对应两种主要回收：

| | Minor GC（新生代 GC） | Full GC（整堆 GC） |
| --- | --- | --- |
| **回收范围** | 只回收新生代（Young / Eden + Survivor） | 整个堆（新生代 + 老年代），可能含元空间 |
| **频率** | 高，几秒到几分钟一次很正常 | 低，理想是几小时甚至几天一次 |
| **单次耗时** | 短，通常几毫秒到几十毫秒 | 长，几百毫秒到几秒，堆大时更久 |
| **STW** | 有，但短 | 有，且长——这是延迟尖刺的主要来源 |
| **健康信号** | 频繁但快，正常 | 越少越好；频繁 Full GC 是危险信号 |

**为什么这样分代有效？** 绝大多数对象是「朝生夕死」的——一个请求进来 `new` 一堆临时对象（DTO、字符串、中间结果），请求处理完就全成垃圾了。这些对象集中在新生代，Minor GC 一扫一大片、回收得又快又干净。少数活得久的对象（缓存、连接池、单例 Bean）会被「晋升」到老年代，老年代回收频率低。

> **前端类比**：Minor GC 像每次组件 re-render 后清掉那些临时变量——高频、轻量、你根本不会注意到。Full GC 像整个 SPA 偶尔做一次彻底的全局内存整理——重，最好别太频繁发生。

### 危险信号：频繁 Full GC

正常服务里 Full GC 应该很稀少。如果你看到 Full GC **几分钟一次甚至几十秒一次**，几乎一定出问题了，常见两种：

1. **内存吃紧 / 配置不当**：堆开得太小（比如容器给了 4G，`-Xmx` 却只设了 512M），对象稍微一多老年代就满，逼着不停 Full GC。
2. **内存泄漏**：有对象只进不出，老年代缓慢被填满，Full GC 每次都收不回多少，过一阵又满——这是 [OOM 与内存泄漏排查](/back-end/frontend-backend-guide/20-oom-memory-leak) 要解决的问题。

判断是哪一种，关键看「**Full GC 之后内存能不能降下来**」：能降下来说明只是堆紧了，加内存或调比例即可；降不下来（每次 Full GC 后老年代占用不断走高）就是泄漏，调参数没用，得去找泄漏点。

## 19.5 常见收集器：以 G1 为主

JDK 这些年出了一堆收集器，名字一大串容易吓人。给前端的实用版认知是：**JDK 9 之后的默认收集器是 G1，你大概率用的就是它，先把 G1 搞懂，其它的知道有这么个东西就行。**

| 收集器 | 一句话定位 | 你需要知道的 |
| --- | --- | --- |
| **G1**（Garbage-First） | JDK 9+ 默认，**分区 + 可控停顿**，通用首选 | 重点掌握这个 |
| **Parallel**（吞吐优先） | JDK 8 默认，追求总吞吐量，不在乎单次停顿长短 | 批处理任务还在用 |
| **CMS** | 老的低延迟收集器，**JDK 14 已彻底移除** | 老项目里可能见到，新项目别用 |
| **ZGC / Shenandoah** | 超低延迟，STW 能压到 1ms 以内，适合大堆 | 几十 G 大堆、延迟极敏感才上 |
| **Serial** | 单线程，最简单，适合小内存 / 客户端 | 服务端基本不用 |

### G1 为什么是默认：分区 + 可控停顿

老式收集器把堆当成「连续的新生代 + 连续的老年代」两大块。G1 不一样：它把整个堆切成很多个大小相等的小块（Region，比如每块 2M），每个 Region 动态扮演 Eden、Survivor 或 Old 的角色。

```text
传统分代：堆是两个连续大块
┌────────────────────┬──────────────────────────────┐
│      新生代          │            老年代              │
└────────────────────┴──────────────────────────────┘

G1：堆被切成一堆小 Region，角色动态分配
┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
│E │O │E │S │O │O │E │  │O │E │S │O │  │E │O │O │
└──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
   E=Eden  S=Survivor  O=Old  空=未分配
   每次只挑「垃圾最多、性价比最高」的几个 Region 回收
```

这样带来两个好处，正好对上后端的诉求：

- **可控停顿**：你可以告诉 G1「我希望每次 STW 别超过 200ms」（参数 `-XX:MaxGCPauseMillis=200`），G1 会**只挑垃圾最多的几个 Region 来收**，凑够时间预算就停，而不是一口气把整个老年代收完。这就是「Garbage-First」——优先收垃圾最多的区域，单位时间回收性价比最高。
- **避免一次性大停顿**：把大回收拆成小批，停顿更可预测。

> **前端类比**：传统收集器像「一次性把整个房间彻底大扫除」，要花的时间不可控。G1 像「定个 20 分钟闹钟，先扫最脏的几个角落，时间到就停，剩下的下次再说」——你能给它一个时间预算，它尽量在预算内把最有价值的活干掉。这对要保证接口延迟稳定的后端正好合适。

调优心法先记一句：**绝大多数情况你不需要换收集器，用默认 G1 + 合理的堆大小就够了。** 换 ZGC、调一堆参数是少数极端场景的事。

## 19.6 实操：开 GC 日志，并读懂它

**目标**：让服务在跑的时候把每一次 GC 都记下来，这样出问题时你能回看「GC 是不是太频繁、停顿是不是太长」。这是排查内存问题的第一手资料，建议**线上默认就开着**（GC 日志开销极小）。

### 第一步：启动时加上 GC 日志参数

JDK 9 及以后用统一的 `-Xlog` 体系：

```bash
# JDK 9+（含 JDK 17，本课程项目用的就是 17）
java -Xlog:gc*:file=/var/log/svc-canvas/gc.log:time,uptime,level,tags:filecount=5,filesize=20M \
     -Xms2g -Xmx2g \
     -XX:MaxGCPauseMillis=200 \
     -jar svc-canvas.jar
```

逐项说明：

- `-Xlog:gc*` —— 打印所有 GC 相关日志（`gc*` 的 `*` 表示 gc 下的所有子标签）。
- `:file=.../gc.log` —— 输出到文件而不是混在标准输出里。
- `:time,uptime,level,tags` —— 每行带上时间戳、运行时长、级别、标签，方便定位。
- `:filecount=5,filesize=20M` —— 日志滚动，最多 5 个文件、每个 20M，防止把磁盘写满。
- `-Xms2g -Xmx2g` —— 初始堆和最大堆都设成 2G（设成一样可以避免堆来回伸缩，生产推荐）。

JDK 8 是另一套老参数（老项目可能还在用）：

```bash
# JDK 8（老项目）
java -XX:+PrintGCDetails -XX:+PrintGCDateStamps \
     -Xloggc:/var/log/gc.log \
     -XX:+UseGCLogFileRotation -XX:NumberOfGCLogFiles=5 -XX:GCLogFileSize=20M \
     -Xms2g -Xmx2g \
     -jar app.jar
```

> **前端类比**：这就像给应用接了一个专门记录「每次内存整理」的 logger，类似你在 Node 里用 `--trace-gc` 启动。平时不看，出问题时它就是案发现场的监控录像。

### 第二步：看一段真实的 GC 日志（JDK 17 / G1）

服务跑起来后，`gc.log` 里会出现这样的内容（节选两次 GC）：

```text
[2026-06-01T10:15:32.110+0800][1203.450s][info][gc,start] GC(142) Pause Young (Normal) (G1 Evacuation Pause)
[2026-06-01T10:15:32.118+0800][1203.458s][info][gc        ] GC(142) Pause Young (Normal) (G1 Evacuation Pause) 1843M->412M(2048M) 8.215ms
[2026-06-01T10:15:32.118+0800][1203.458s][info][gc,cpu    ] GC(142) User=0.04s Sys=0.00s Real=0.01s

[2026-06-01T10:18:05.642+0800][1356.982s][info][gc,start] GC(187) Pause Full (G1 Compaction Pause)
[2026-06-01T10:18:06.531+0800][1357.871s][info][gc        ] GC(187) Pause Full (G1 Compaction Pause) 1987M->1502M(2048M) 889.331ms
[2026-06-01T10:18:06.531+0800][1357.871s][info][gc,cpu    ] GC(187) User=2.10s Sys=0.05s Real=0.89s
```

### 第三步：怎么读这段日志

挑出关键那一行逐字段拆开看。先看第一次（健康的 Minor GC）：

```text
GC(142) Pause Young (Normal) (G1 Evacuation Pause) 1843M->412M(2048M) 8.215ms
   │         │                                        │     │   │       │
   │         │                                        │     │   │       └─ 本次 STW 耗时：8.2 毫秒
   │         │                                        │     │   └─ 堆总大小：2048M
   │         │                                        │     └─ 回收「后」堆占用：412M
   │         │                                        └─ 回收「前」堆占用：1843M
   │         └─ 类型 Pause Young = Minor GC（只收新生代）
   └─ 第 142 次 GC（编号递增，可数频率）
```

怎么解读：堆从 1843M 一下降到 412M，**回收掉了约 1.4G 垃圾**——说明新生代里全是用完即弃的临时对象，回收得很干净。耗时只有 **8.2ms**，肉眼无感。`gc,cpu` 那行 `Real=0.01s` 是真实墙上时间。**这是一次教科书级健康的 Minor GC。**

再看第二次（一次值得警惕的 Full GC）：

```text
GC(187) Pause Full (G1 Compaction Pause) 1987M->1502M(2048M) 889.331ms
            │                              │      │              │
            │                              │      │              └─ STW 耗时：889 毫秒！接近 1 秒
            │                              │      └─ 回收后仍占 1502M（堆共 2048M）
            │                              └─ 回收前 1987M（已经快满了）
            └─ 类型 Pause Full = Full GC（整堆回收）
```

怎么解读，三个红灯：

1. **触发了 Full GC**：回收前堆已到 1987M / 2048M，几乎满了，被逼着做整堆回收。
2. **STW 长达 889ms**：这一下，`svc-canvas` 上所有请求被卡了将近 1 秒。监控上会看到这一刻 P99 延迟飙到 900ms+，用户感觉「卡了一下」。
3. **回收后还剩 1502M**：整堆扫了一遍才降到 1502M，**只回收了不到 500M**。说明堆里有大量「活着、回收不掉」的对象长期占着老年代——要么堆太小该加内存，要么有内存泄漏。

### 判断「GC 是否健康」的经验清单

不用记参数，记这三条经验，对着 GC 日志（或 Grafana 的 GC 面板）扫一眼就能下判断：

| 看什么 | 健康 | 不健康（该查了） |
| --- | --- | --- |
| **Full GC 频率** | 几小时一次，甚至更久 | 几分钟一次、几十秒一次 |
| **单次 STW 时长** | Minor 几十 ms 内；Full 控制在停顿目标内 | 单次几百 ms ~ 几秒，且频繁 |
| **回收后内存** | Full GC 后老年代明显降下来 | 每次 Full GC 后内存降不下去、整体水位持续走高 |

最关键的是第三条——**Full GC 后内存降不下来**，基本就是内存泄漏的铁证，这时候别折腾 GC 参数，直接转 [OOM 与内存泄漏排查](/back-end/frontend-backend-guide/20-oom-memory-leak) 去抓泄漏点（dump 堆、看对象直方图）。

> **前端类比**：这就像你在 Chrome Performance 面板看到一连串高高的 GC 紫色条、或者在 Memory 面板里反复 Snapshot 发现某类对象数量只涨不跌——只涨不跌就是泄漏。后端用 GC 日志看的是同一件事，只是工具不同。

## 19.7 调优心法：先别动参数

新手一看到「GC 频繁」「接口卡顿」，第一反应往往是上网搜一堆 `-XX:` 参数往上堆。**这通常是错的，而且会越调越乱。** 正确的顺序是：

```text
GC 异常（频繁 Full GC / STW 长）
        │
        ▼
① 先看日志确认现象：是 Full GC 频繁，还是单次 STW 长？回收后内存降不降？
        │
        ▼
② 排查是不是「内存泄漏」（回收后内存降不下来）
        │  是泄漏 → 去 dump 堆、找泄漏点（第 20 章），调参数无用
        ▼  不是泄漏，往下
③ 排查是不是「配置不当」（堆太小、容器内存与 -Xmx 不匹配）
        │  是配置问题 → 把 -Xmx 调到容器内存的合理比例，常常一调就好
        ▼  配置也合理，往下
④ 才轮到「真正的 GC 调优」（调停顿目标、新生代比例、换收集器）
        └─ 这是少数情况，且要基于数据，一次只改一个参数再观测
```

一句话心法：**先排查是不是泄漏或配置不当，确认这两样都没问题，再考虑动 GC 参数；动参数也要一次只改一个、改完看日志对比，别一把梭。** 真实生产里，绝大多数「GC 问题」最后查出来不是 GC 本身的问题，而是泄漏或者 `-Xmx` 设错了。

> **前端类比**：这跟「页面卡了别急着上 `useMemo`/`useCallback` 乱包一通」是一个道理——先用 Profiler 定位到底是哪个组件、哪次渲染慢，找到真因再针对性优化。瞎优化往往制造新问题。GC 调优更是如此，因为参数之间互相影响，乱调很容易顾此失彼。

## 小结

- **GC 的职责**是自动回收「再也用不到的对象」，让你只管 `new`、不用手动 `free`——和 JS 引擎的 GC 同源，区别是 JVM 的 GC 可观测、可调、能看日志。
- **判断对象死活靠可达性分析**：从 GC Roots（栈变量、static 字段等）出发够不到的对象就是垃圾。它能正确处理「循环引用」，这是老式引用计数做不到的。
- **STW（Stop The World）是后端必须关心 GC 的根本原因**：回收时要暂停全部业务线程，STW 太长 = 这个进程上所有用户的所有接口同时卡顿。
- **Minor GC 频繁但快、可接受；Full GC 慢且要尽量少**。频繁 Full GC 通常意味着内存吃紧或内存泄漏——判别关键看「Full GC 后内存能不能降下来」。
- **G1 是 JDK 9+ 默认收集器**，靠分区 + 可控停顿（`MaxGCPauseMillis`）平衡「收干净」与「停顿短」；CMS 已废弃，ZGC/Shenandoah 是大堆超低延迟才上的选项。
- **调优心法：先排查泄漏和配置，别一上来瞎调参数**；动参数也要基于 GC 日志数据、一次只改一个。

### 自测

1. 有两个对象 `a` 和 `b` 互相引用，但已经没有任何变量指向它们了。用「引用计数」和「可达性分析」分别判断，它们能不能被回收？为什么 JVM 选了后者？
2. `svc-canvas` 监控显示每隔几十秒就出现一次接近 1 秒的延迟尖刺，所有接口同时变慢。结合 STW 和 Full GC 解释这个现象最可能的成因，以及你会先去看哪份日志、重点看哪几个字段。
3. 给你一行 GC 日志：`Pause Full (G1 Compaction Pause) 1987M->1502M(2048M) 889.331ms`。请说出这是哪种 GC、回收前后堆占用各是多少、STW 多久、以及这次 GC 你觉得健康吗、下一步该往哪个方向排查？
