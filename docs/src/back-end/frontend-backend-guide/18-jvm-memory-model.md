# JVM 内存模型

> 前端排查内存，你打开 Chrome DevTools 的 Memory 面板，拍一张 Heap Snapshot，看哪些对象没被回收。Java 后端也有堆、有调用栈、有 GC——思路相通。区别是：V8 的内存细节大多是黑盒、你管不着；JVM 把内存切成几块明明白白的「运行时数据区」，而且**允许你用启动参数手动调**。
>
> 这一章不解决问题，它是给第 19 章（GC 怎么回收）和第 20 章（OOM/内存泄漏怎么排查）打地基。你只有先知道「内存被分成了哪几块、对象住在哪、引用在哪」，后面那两章的 `jmap`、`-Xmx`、`OutOfMemoryError: Java heap space` 才不是天书。

读这一章时心里装一个场景：`svc-canvas` 跑在 K8s 的一个 Pod 里，容器限制 2GB 内存。它每秒处理几十个生图任务，连续跑了两周没重启。这两周里它的内存是怎么用的、为什么会涨、涨到顶会发生什么——答案全在 JVM 内存模型里。

## 18.1 先建立总图：进程内存被切成了几块

一个 Java 进程（比如一个 `svc-ai` 实例）启动后，JVM 会把它能用的内存切分成几个**运行时数据区（Runtime Data Areas）**。有的全进程共享，有的每个线程一份。

```text
┌───────────────────────────────────────────────────────────────┐
│                  一个 JVM 进程（如 svc-canvas）                  │
│                                                                 │
│   ┌──────────────────────── 所有线程共享 ────────────────────┐  │
│   │                                                          │  │
│   │   堆 Heap（GC 主战场，-Xmx 控制大小）                      │  │
│   │   ┌──────────── 新生代 Young ────────────┐ ┌─ 老年代 ──┐ │  │
│   │   │  Eden        │ S0(From) │ S1(To)     │ │  Old      │ │  │
│   │   │ 新对象在这    │  幸存区   │  幸存区     │ │ 长命对象   │ │  │
│   │   └──────────────────────────────────────┘ └──────────┘ │  │
│   │                                                          │  │
│   │   方法区 / Metaspace（类元数据，JDK8 起在本地内存）         │  │
│   │   存：类信息、方法字节码、运行时常量池、static 字段引用      │  │
│   └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│   ┌──────────── 每个线程各一份（线程私有）────────────────────┐  │
│   │  线程 1            线程 2            线程 200 (Tomcat worker)│  │
│   │  ┌──────────┐     ┌──────────┐     ┌──────────┐           │  │
│   │  │虚拟机栈   │     │虚拟机栈   │     │虚拟机栈   │  栈帧叠栈帧 │  │
│   │  │  栈帧     │     │  栈帧     │     │  栈帧     │           │  │
│   │  ├──────────┤     ├──────────┤     ├──────────┤           │  │
│   │  │程序计数器 │     │程序计数器 │     │程序计数器 │  当前执行行 │  │
│   │  ├──────────┤     ├──────────┤     ├──────────┤           │  │
│   │  │本地方法栈 │     │本地方法栈 │     │本地方法栈 │  调 native │  │
│   │  └──────────┘     └──────────┘     └──────────┘           │  │
│   └──────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────┘
```

记住这条分界线，后面所有细节都挂在它上面：

- **共享区**（堆、方法区/Metaspace）：所有线程都能看到同一份数据，所以这里的数据有「多线程同时改」的安全问题（第 16 章），也是 GC 要清理的对象（第 19 章）。
- **私有区**（虚拟机栈、程序计数器、本地方法栈）：每个线程开一份，互不干扰，线程死了就回收，**不需要 GC、也不会有并发问题**。

> **前端类比**：V8 也是这个二分结构——Heap（对象都在这）+ Call Stack（函数调用一层层压栈）。你在 DevTools 里看到的 "Memory" 面板对应 Java 的堆，"Call Stack" 面板对应虚拟机栈。最大的不同：JS 是单线程，所以只有一个调用栈；Java 一个进程里有几百个线程，就有**几百个栈同时存在**。

## 18.2 逐块拆解

### 堆 Heap：对象的家，GC 的战场

**所有用 `new` 创建的对象实例、所有数组，都住在堆里。** 堆是所有线程共享的，也是整个 JVM 里最大的一块，是 GC 唯一真正盯着回收的地方。

`svc-ai` 里每来一个生图请求，你 `new` 出来的 `GenImageReq`、查出来的 `User` 对象、拼出来的 `RtData`——全在堆上。请求处理完、没人再引用它们，它们就成了垃圾，等着被 GC 清走。

堆的大小由启动参数 `-Xms`（初始）和 `-Xmx`（最大）控制。**堆满了又回收不出空间，就抛 `OutOfMemoryError: Java heap space`**——这是后端最常见的 OOM，第 20 章专门讲怎么排。

> **前端类比**：就是 V8 的 Heap。你 `const obj = {}` 对象进堆，没人引用了等 GC（V8 的标记清除）回收。区别是 V8 堆大小你基本调不动（默认上限约 1.4~2GB），而 Java 你能用 `-Xmx4g` 明确指定。

### 虚拟机栈 Java Virtual Machine Stack：方法调用的轨迹

**每个线程启动时配一个虚拟机栈。** 线程每调用一个方法，就压入一个**栈帧（Stack Frame）**；方法返回，栈帧弹出。一个栈帧里装着这个方法的：

- **局部变量表**：方法里的参数和局部变量（基本类型的值、对象的引用地址）；
- 操作数栈、动态链接、方法返回地址等（执行用的临时数据）。

调用层级越深，栈帧叠得越高。如果递归没有终止条件无限调下去，栈就会撑爆，抛 **`StackOverflowError`**——这跟堆 OOM 是两回事，别搞混。

```text
svc-canvas 处理一个请求时，某线程的虚拟机栈（自顶向下 = 当前往回看）：

   ┌─────────────────────────────┐  ← 栈顶（当前正在执行）
   │ 栈帧: ossClient.upload(...)  │
   ├─────────────────────────────┤
   │ 栈帧: aiClient.generate(req) │
   ├─────────────────────────────┤
   │ 栈帧: CanvasService.submit() │
   ├─────────────────────────────┤
   │ 栈帧: CanvasController.gen() │
   └─────────────────────────────┘  ← 栈底（最先入栈）
```

> **前端类比**：这就是你在 DevTools 里看到的 **Call Stack**，也是 JS 报错时那一长串 stack trace。`a() → b() → c()` 一层层压栈，`c` 返回再弹回 `b`。完全一样。Java 报异常打出来的 `at com.xxx.CanvasService.submit(...)` 这一行行，就是把当时的栈帧从上到下打印出来。

### 程序计数器 Program Counter Register：当前执行到第几行

**每个线程一个，记录这个线程当前执行到哪条字节码指令。** 你可以把它理解成线程的「书签」。

为什么需要它？因为后端是多线程，一个线程执行到一半被 CPU 切走，让别的线程先跑；轮回来时，靠这个计数器知道「我上次执行到哪了，从这接着跑」。它很小，不会 OOM，几乎不用你操心。

> **前端类比**：JS 单线程，永远只有一条执行流，所以你从没听说过「程序计数器」。但 Java 几百个线程要轮流用 CPU，每个线程都得记住自己的进度——这是多线程世界才需要的东西。

### 本地方法栈 Native Method Stack：调用非 Java 代码时用

跟虚拟机栈类似，但服务于 **native 方法**（用 C/C++ 写的、JVM 底层的方法，比如某些 IO、加密操作）。日常业务代码基本不用关心它，知道有这么个东西即可。

### 方法区 / Metaspace：类的「说明书」放哪

**方法区存的是「类」级别的元数据**，不是对象实例：每个类长什么样（字段、方法）、方法的字节码、运行时常量池、`static` 静态变量的引用等。一个类只加载一次，这份元数据就常驻在这里。

关键变化：**JDK 8 之前这块叫「永久代 PermGen」，在堆里**；**JDK 8 起改名 Metaspace，搬到了「本地内存」（堆外，操作系统直接管的那部分）**。我们项目是 Java 17，所以是 Metaspace，不在 `-Xmx` 管的堆里。

这块也会 OOM：如果程序不停动态生成类（比如某些框架滥用动态代理、热部署反复加载），Metaspace 会涨，抛 `OutOfMemoryError: Metaspace`。它的大小不归 `-Xmx` 管，要用 `-XX:MetaspaceSize` / `-XX:MaxMetaspaceSize` 单独设。

> **前端类比**：勉强类比的话，Metaspace 像浏览器解析、编译后缓存的「模块定义/类定义」本身，而堆是按这些定义 `new` 出来的实例。定义只有一份，实例可以有千千万。前端没有显式的「类元数据区」概念，这是 JVM 特有的。

## 18.3 对象在堆，引用在栈：一段代码标住每个变量住哪

这是最容易绕晕、也最值得彻底搞清的一点。看 `svc-user` 里一段扣配额的代码，逐个标注：

```java
public RtData<Integer> deductQuota(Long userId, int cost) {
    // userId 形参：引用，在【栈】（指向某个 Long 对象，对象在堆）
    // cost  形参：基本类型 int，值直接存在【栈】的局部变量表里

    int remaining = 100;          // remaining: int 基本类型，值在【栈】
    User user = userRepo.find(userId);
    // user：是个【引用】，存在【栈】里；
    // new 出来的那个 User 实例（含它的所有字段）在【堆】里。
    // 栈里的 user 只是一个"地址"，指向堆里那块对象。

    List<String> tags = new ArrayList<>();
    // tags：引用在【栈】；ArrayList 对象本体、以及它内部装的元素都在【堆】

    user.setQuota(user.getQuota() - cost);   // 改的是【堆】里那个 User 的字段
    return RtData.ok(user.getQuota());
}   // 方法返回 → 这个栈帧整个弹出 → remaining/user/tags 这些"栈上的引用"全没了
    // 但堆里的 User 对象不会立刻消失——要等 GC 确认"再没有任何引用指向它"才回收
```

一句话总结：**基本类型（int/long/boolean…）的值直接放栈上；对象一律放堆，栈上只放一个指向它的引用（地址）。**

```text
        栈（线程私有，方法返回即弹出）        堆（共享，等 GC 回收）
        ┌────────────────────┐            ┌──────────────────────────┐
        │ cost      = 5       │            │  User@a1b2 {              │
        │ remaining = 100     │            │     id: 1001,             │
        │ userId  ──────────┐ │            │     quota: 95,            │
        │ user    ────────┐ │ └──────────▶ │     name: "alice"         │
        │ tags    ──────┐ │ │              │  }                        │
        └───────────────│─│─│──────────────┤                          │
                        │ │ └────────────▶ │  Long@c3d4 { 1001 }       │
                        │ └──────────────▶ │  ArrayList@e5f6 { ... }   │
                        └────────────────▶ │  (tags 指向的列表对象)     │
                                           └──────────────────────────┘
```

这解释了两个后端高频现象：

1. **为什么方法返回了，对象不一定马上被回收**：栈上的引用没了，但只要堆里那个对象还被别的地方引用着（比如塞进了一个 `static` 的缓存 Map），它就活着。这正是**内存泄漏**的根源——见第 20 章。
2. **为什么多线程改同一个对象会出事**：堆是共享的。线程 A 和线程 B 的栈各自有一个 `user` 引用，但可能指向**堆里同一个对象**，于是两个线程同时改它就会互相踩踏（第 16 章的线程安全问题）。

> **前端类比**：JS 里 `const user = { quota: 100 }`，变量 `user` 在栈/作用域里只是引用，对象 `{...}` 在堆里——和 Java 一模一样。你传 `user` 给另一个函数改它的属性，外面也变了（引用传递），就是因为大家指向堆里同一个对象。Java 的对象传参也是这个机制。

## 18.4 堆为什么要分代：朝生夕死假说

堆不是一整块平铺的内存，JVM 把它分成**新生代（Young Generation）**和**老年代（Old Generation）**。新生代又细分成一个 **Eden 区**和两个 **Survivor 区（S0/S1，也叫 From/To）**。

为什么这么折腾？因为有一条被无数实践验证的经验规律——**「朝生夕死」假说（Weak Generational Hypothesis）**：

> **绝大多数对象都活不过一次 GC**，只有极少数对象会长期存活。

放到 `svc-canvas` 里特别直观：每个生图请求里 `new` 出来的 `GenImageReq`、临时拼的字符串、查出来的临时 DTO——请求一处理完就成垃圾了，活不过几毫秒。而像 Spring 容器里的单例 Bean、连接池对象、本地缓存，则从启动活到关机。

既然「死得快的多、活得久的少」，分代就能让 GC 高效得多：

```text
对象的一生（在堆里搬家的过程）：

   new 一个对象
        │
        ▼
   ┌─────────┐  Eden 满了，触发 Minor GC
   │  Eden   │  ──────────────────────────┐
   └─────────┘                            │ 大多数对象在这里就被回收了（朝生夕死）
        │ 活下来的少数                      │
        ▼                                 ▼
   ┌─────────┐  每熬过一次 Minor GC      （成为垃圾，被清走）
   │Survivor │  年龄 +1，在 S0/S1 间来回搬
   └─────────┘
        │ 年龄到阈值（默认 15）/ 一直活着
        ▼
   ┌─────────┐  老年代满了才触发 Major/Full GC
   │  Old    │  （回收成本高、停顿更长，要尽量少触发）
   └─────────┘
```

几条要记住的规则：

- **新对象先进 Eden。** Eden 满了，触发一次 **Minor GC（只清新生代，很快）**，把还活着的对象挪到 Survivor，绝大部分对象在这一步就被清掉了。
- **每熬过一次 Minor GC，对象年龄 +1**，并在两个 Survivor 区之间来回倒。年龄到阈值（默认 15，`-XX:MaxTenuringThreshold`）还活着，就**晋升到老年代**——它被认定为「长命对象」。
- **大对象直接进老年代。** 一个很大的数组/对象（超过 `-XX:PretenureSizeThreshold`）放不进 Eden，干脆直接扔老年代，免得在新生代来回搬。比如一次性 `new byte[10*1024*1024]` 读一张大图。
- **老年代满了才触发 Full GC**，它要扫描整个堆，停顿（STW）明显更长。线上「接口偶发性卡顿几百毫秒甚至几秒」，常常就是 Full GC 干的。

> **前端类比**：V8 也是分代的——新对象进 "New Space"（新生代），存活久的晋升到 "Old Space"（老年代），同样基于「大部分对象很快就死」的假说。所以这套分代思想你其实早就在用，只是 V8 在背后默默做、不让你调；Java 把每一代的大小、晋升阈值都开放成参数让你调。第 19 章会专门讲这两代分别用什么 GC 算法回收。

## 18.5 关键启动参数：怎么给 JVM 划内存

这些参数会出现在你的启动脚本或 K8s 的 `JAVA_OPTS` 里。先认识最核心的几个：

| 参数 | 含义 | 举例 |
| --- | --- | --- |
| `-Xms` | 堆的**初始**大小 | `-Xms2g` 启动就占 2GB 堆 |
| `-Xmx` | 堆的**最大**大小 | `-Xmx2g` 堆最多涨到 2GB，再满就 OOM |
| `-Xmn` | **新生代**大小 | `-Xmn768m` 新生代占 768MB |
| `-XX:MetaspaceSize` | Metaspace 初始触发 GC 的阈值 | `-XX:MetaspaceSize=128m` |
| `-XX:MaxMetaspaceSize` | Metaspace 上限（不设则受本地内存限制） | `-XX:MaxMetaspaceSize=256m` |
| `-XX:+HeapDumpOnOutOfMemoryError` | OOM 时自动 dump 堆，供事后分析 | 配合 `-XX:HeapDumpPath=/dump` |

一个 `svc-ai` 的典型启动命令：

```bash
java -Xms2g -Xmx2g -Xmn768m \
     -XX:MetaspaceSize=128m -XX:MaxMetaspaceSize=256m \
     -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/dump/svc-ai.hprof \
     -jar svc-ai.jar
```

> **实操经验：为什么生产上常把 `-Xms` 和 `-Xmx` 设成一样？**
> 如果初始小、最大大，JVM 跑着跑着发现内存不够会动态扩堆，扩堆过程伴随 GC 和内存重排，会引起额外停顿。把两者设成相等，启动时就一次性占满，运行期不再伸缩，停顿更平稳可预测。生产环境几乎都这么干。

怎么确认参数生效、当前堆用了多少？用 `jstat` 看一眼（第 29 章诊断工具箱会展开，这里先尝个鲜）：

```bash
# 目标：看 svc-ai 进程各内存区的使用率，每 2 秒打印一次
jps                          # 先找出 Java 进程 PID
jstat -gcutil <pid> 2000
```

预期输出（每行一次采样）：

```text
  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
  0.00  31.25  68.40  45.12  96.30  92.10    128    1.245      3    0.890   2.135
```

怎么读这段输出：

- `E`（Eden）`68.40` = Eden 用了 68%，正常涨；涨到 100 就触发一次 Minor GC，`E` 掉回低位、`YGC`（Young GC 次数）+1。
- `O`（Old）`45.12` = 老年代用了 45%。如果你盯着看它**只涨不降、逼近 100%**，又伴随 `FGC`（Full GC 次数）不停增加——这就是内存泄漏的典型信号，对象一直晋升进老年代却回收不掉。
- `YGC=128 / FGC=3`：启动到现在做了 128 次 Minor GC、3 次 Full GC。`FGC` 频繁是危险信号。
- `M`（Metaspace）`96.30` 已经很高了，如果还在涨，警惕 `OutOfMemoryError: Metaspace`。

**结论**：`-gcutil` 是你判断「内存到底健不健康」的第一眼工具——`O` 稳定波动 = 健康；`O` 阶梯式只涨不回 + `FGC` 频繁 = 要去查内存泄漏了（第 20 章）。

## 18.6 容器里的大坑：不按容器内存设堆，会被 OOMKilled

这是前端转后端、第一次把服务塞进 Docker/K8s 时几乎必踩的坑，务必记牢。

**症状**：`svc-canvas` 容器内存 limit 设了 2GB，但 Pod 频繁重启，`kubectl describe pod` 里写着 `Reason: OOMKilled`、`Exit Code: 137`。诡异的是，应用日志里**根本没有 `OutOfMemoryError`**——它不是 Java 自己抛的 OOM，是被操作系统/K8s 直接杀的。

**根因**：JVM 默认会按「**它看到的物理内存**」的一个比例（约 1/4）去推算最大堆。在老版本 JDK 或没开容器感知时，JVM 可能看到的是**整台宿主机的内存（比如 64GB）**，而不是容器 limit 的 2GB，于是它把堆上限算成了 16GB。结果堆还没满（Java 自己觉得没事），进程实际占用就已经超过容器 limit 的 2GB，被 K8s 一刀 `OOMKilled`。

```text
错误的认知                          实际发生的
─────────────────                   ─────────────────────────
JVM 以为可用 = 宿主机 64GB           容器 limit = 2GB
   → 把堆上限自动算成 ~16GB           → 进程一超过 2GB
   → Java 觉得堆还很空                → 内核/K8s 立刻 OOMKilled（137）
   → 不抛 OutOfMemoryError            → 日志里看不到任何 Java 异常 ← 迷惑点
```

**怎么修**——两选一：

```bash
# 方案 A：显式按容器内存写死堆大小（最直白，强烈推荐）
# 容器 2GB，给堆留 1.2~1.5g，剩下留给 Metaspace/线程栈/堆外/JVM 自身
java -Xms1500m -Xmx1500m -jar svc-canvas.jar
```

```bash
# 方案 B：让 JVM 感知容器（JDK 10+ 默认开启容器感知；用比例而非写死值）
# 堆 = 容器 limit 的 75%，由 JVM 自动按容器 limit（不是宿主机）计算
java -XX:MaxRAMPercentage=75.0 -jar svc-canvas.jar
```

> **关键提醒**：注意「容器内存」≠「堆」。容器 limit 2GB 里，**堆只是其中一部分**，还要分给 Metaspace、每个线程的虚拟机栈（默认每线程约 1MB，200 个 Tomcat 线程就是约 200MB）、堆外内存、JVM 自身开销。所以堆千万别设到逼近容器 limit，得留出余量，否则即使堆没满，进程总占用也会顶破 limit 被杀。这个坑和 K8s 的资源限制紧密相关，详见 [Kubernetes 实战](/back-end/frontend-backend-guide/24-kubernetes-in-practice)。

> **前端类比**：有点像你给 Docker 里的 Node 应用配 `--max-old-space-size`——如果不配，Node 也可能按宿主机内存估算堆上限，在受限容器里被 OOMKilled。Java 这边问题更典型，因为 JVM 默认堆比例算法更激进、内存区也更多，留余量这件事更要上心。

## 小结

- 一个 JVM 进程的内存切成几块：**堆**（对象实例、所有线程共享、GC 主战场）、**虚拟机栈**（每线程一份、栈帧=方法调用、存局部变量和引用）、**方法区/Metaspace**（类元数据，JDK 8 起搬到本地内存）、**程序计数器**、**本地方法栈**。共享区有并发与 GC 问题，私有区随线程消亡而回收。
- **对象在堆，引用在栈**：基本类型的值直接存栈，对象一律在堆、栈里只放指向它的引用。方法返回栈帧弹出，但堆对象要等「再无引用」时才被 GC 回收——这是内存泄漏的根源。
- **堆分代**基于「朝生夕死」假说：新对象进 Eden，熬过 Minor GC 进 Survivor，年龄够了或太大就晋升老年代；老年代满才触发更慢的 Full GC，是线上偶发卡顿的常见来源。
- **关键参数**：`-Xms/-Xmx`（堆初始/最大，生产常设相等）、`-Xmn`（新生代）、`-XX:MetaspaceSize/-XX:MaxMetaspaceSize`（Metaspace，不归 `-Xmx` 管）。`jstat -gcutil` 是判断内存健康的第一眼工具。
- **容器里必须按容器内存设堆**，否则 JVM 可能按宿主机内存把堆算得过大，进程超过容器 limit 被 `OOMKilled`（Exit Code 137，且日志里看不到 Java 的 `OutOfMemoryError`）。堆要给 Metaspace、线程栈、堆外留余量，别顶满 limit。

### 自测

1. `svc-user` 的 `deductQuota(Long userId, int cost)` 方法里，`cost`、`userId`、`new ArrayList<>()` 创建的列表，分别存在哪个内存区？方法返回后，谁会立刻消失、谁不会，为什么？
2. 「朝生夕死」假说说的是什么？基于它，为什么把堆分成新生代和老年代能让 GC 更高效？一个对象从 `new` 出来到进入老年代，要经历哪些步骤？
