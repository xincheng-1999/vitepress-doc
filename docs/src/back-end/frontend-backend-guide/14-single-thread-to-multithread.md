# 从 JS 单线程到 Java 多线程

> 这是 Part 4「并发」的开篇，也是整门课**认知转变最大**的一章。
> 你过去写前端，从没担心过「两段代码同时改一个变量」——因为浏览器根本不会让这种事发生。
> 但到了后端，这恰恰是每天都要面对的现实。读懂这一章，你才算真正踏进后端的大门。

本章只负责**建立认知**：搞清楚为什么后端会有「线程安全」这个全新维度。真正的解决方案在 [第十五章 线程池](/back-end/frontend-backend-guide/15-thread-pools-executor) 和 [第十六章 线程安全与锁](/back-end/frontend-backend-guide/16-thread-safety-locks) 展开。

## 14.1 先回顾你最熟的：JS 单线程模型

你写了一年多前端，但可能从没认真想过：**为什么 JS 里从来不需要加锁？**

答案是——JavaScript 在一个页面里只有**一个主线程**。它配了一个**事件循环（Event Loop）**和若干**任务队列**，所有你的代码（同步代码、`setTimeout` 回调、Promise 的 `.then`、`await` 之后的代码）都排成队，一段执行完了，才轮到下一段。任何时刻，**永远只有一段 JS 代码在跑**。

```text
        ┌─────────────────────────────────────────┐
        │              调用栈 Call Stack            │
        │   （任意时刻只跑这里的一段代码，单线程）  │
        └───────────────────▲─────────────────────┘
                            │ 取出一个任务执行
                            │
   ┌────────────────────────┴───────────────────────────┐
   │                   Event Loop                        │
   │      栈空了 → 从队列取下一个任务放进栈               │
   └───▲──────────────────────────────▲─────────────────┘
       │                              │
 ┌─────┴───────┐              ┌───────┴──────────┐
 │ 宏任务队列  │              │  微任务队列      │
 │ setTimeout  │              │  Promise.then    │
 │ DOM 事件    │              │  await 之后      │
 └─────────────┘              └──────────────────┘
```

关键认知：在前端写下面这段代码，你**100% 确信**结果是 `2`：

```javascript
let balance = 0

function addOne() {
  const current = balance // 读
  balance = current + 1 // 写
}

addOne()
addOne()
console.log(balance) // 一定是 2，从来不会是 1
```

为什么这么笃定？因为第一个 `addOne()` 必须**完整跑完**（读 + 写都做完），第二个 `addOne()` 才会开始。它们之间不可能「插队」。哪怕是异步代码，`await` 也只是把后面的代码**排到队列里晚点跑**，而不是「另开一个线程同时跑」。

> 💡 **前端类比**：单线程就像一条单行道上只有一辆车。无论红绿灯怎么切换（事件循环调度），任何瞬间路上只有这一辆车在动，所以两辆车永远不会撞——因为压根没有第二辆。前端的「并发」是**靠快速切换任务制造的错觉（concurrency）**，不是真的同时跑。

## 14.2 后端的现实：一个进程里跑着一大群线程

把同样的逻辑搬到后端，世界观就变了。

一个 Java 后端服务（比如我们的 `svc-user`）启动后是一个 **JVM 进程**。这个进程内部**同时活跃着很多线程**：处理 HTTP 请求的、跑定时任务的、消费 RocketMQ 消息的、垃圾回收的……它们是**真正并行**的——在多核 CPU 上，是物理意义上的「同一时刻好几段代码都在动」。

最关键的来源是 **Web 容器的线程池**。Spring Boot 内嵌的 Tomcat 默认维护一个约 200 个线程的池子，**每来一个 HTTP 请求，就分配一个线程去处理它**：

```text
                       svc-user（一个 JVM 进程）
   ┌──────────────────────────────────────────────────────────┐
   │                  Tomcat 线程池（约 200 个）                │
   │                                                            │
   │  请求 #1 ─▶ thread-http-1 ──┐                              │
   │  请求 #2 ─▶ thread-http-2 ──┤                              │
   │  请求 #3 ─▶ thread-http-3 ──┤  这些线程同时执行            │
   │   ...                       ├─▶ 同一份 UserService 代码    │
   │  请求 #N ─▶ thread-http-N ──┘                              │
   │                                                            │
   │  另外还有: RocketMQ 消费线程 / xxl-job 定时任务线程        │
   │            GC 线程 / 各种连接池后台线程 ...                │
   └──────────────────────────────────────────────────────────┘
```

注意框图里那句话：**这些线程同时执行同一份 `UserService` 代码**。

这是和前端最本质的区别。前端一个页面同一时刻只有一个「你」在跑代码；后端一个服务同一时刻有几十上百个「分身」在跑**同一个方法**。如果这个方法里读写了某个**共享的数据**，多个分身就会**抢着读、抢着写**。

> 💡 **前端类比**：如果说 Node.js 单进程像「一个收银员，再忙也是一个一个结账」，那 Tomcat 线程池就像「一家超市开了 200 个收银台同时结账」。收银台多了效率高，但如果这 200 个收银台**共用一本账本**（共享变量），就会出现两个收银员同时翻到同一页、同时改、互相覆盖的混乱——这正是后端要解决的核心问题。

## 14.3 翻车现场：两个线程同时给余额 +1

把 14.1 那段「一定是 2」的 JS 代码，翻译成会被多线程同时调用的 Java 代码。假设 `svc-user` 里有个充值逻辑，给用户账户余额加 1：

```java
// 一个被多个请求线程共享的对象（注意：单例！全 JVM 只有这一个 account）
public class Account {
    private int balance = 0; // 共享变量

    public void addOne() {
        int current = balance; // ① 读
        // 想象这里有一丁点耗时（比如 CPU 调度切走了）
        balance = current + 1; // ② 写
    }

    public int getBalance() {
        return balance;
    }
}
```

现在开两个线程，各调用一次 `addOne()`：

```java
public class RaceDemo {
    public static void main(String[] args) throws InterruptedException {
        Account account = new Account();

        // 让两个线程各 +1，循环 100000 次放大问题
        Runnable task = () -> {
            for (int i = 0; i < 100_000; i++) {
                account.addOne();
            }
        };

        Thread t1 = new Thread(task, "worker-1");
        Thread t2 = new Thread(task, "worker-2");

        t1.start();
        t2.start();
        t1.join(); // 等 t1 跑完
        t2.join(); // 等 t2 跑完

        System.out.println("期望值: 200000, 实际值: " + account.getBalance());
    }
}
```

**预期输出（多跑几次，每次都不一样）：**

```text
期望值: 200000, 实际值: 173852
期望值: 200000, 实际值: 168930
期望值: 200000, 实际值: 200000   ← 偶尔也会蒙对
```

**怎么读这段输出**：结果几乎从不等于 200000，而且**每次都不同**。这就是「竞态条件（Race Condition）」最典型的症状——**结果不可复现，时灵时不灵**。

问题出在哪？`addOne()` 看起来是一行 `balance = balance + 1`，但它在 CPU 眼里其实是**三步**：读 → 加 → 写。两个线程真正并行时，可能这样交错：

```text
时间 ──────────────────────────────────────────▶

worker-1:   读 balance=5 ──────────▶ 写 balance=6
                       ╲          ╱
                        ╲ 交错！ ╱
                       ╱          ╲
worker-2:        读 balance=5 ──────────▶ 写 balance=6

结果：两个线程各 +1，余额却只从 5 变成 6（少加了一次！）
```

两个线程都读到了 `5`，各自算出 `6`，先后写回——**后写的覆盖了先写的，一次自增凭空消失了**。在 10 万次循环里，这种「丢失更新」会累积成几万的误差。

> 💡 **为什么前端写一辈子都不会遇到这个？** 因为 JS 单线程保证了 `addOne()` 永远是**原子**地跑完（读、加、写连在一起没人能插进来）。而 Java 多线程没有这个保证。**这就是「线程安全」问题——一个前端世界里根本不存在的全新维度。**

在我们的 AI 生图项目里，这类场景遍地都是：

- `svc-user` 给用户**扣配额**：两个生图请求同时进来，都读到「剩余 1 次」，都判断「够」，都扣减——结果配额扣成了 -1，用户白嫖了一次。
- `svc-canvas` 统计**任务计数**：多个消费线程同时给「今日生成总数」自增，数字越统计越少。

解决办法（加锁、用原子类、用 Redis 分布式锁）是 [第十六章](/back-end/frontend-backend-guide/16-thread-safety-locks) 的主题。本章你只需要**记住这个坑的存在**。

## 14.4 概念澄清：进程 vs 线程

| 概念 | 是什么 | 在我们的项目里 | 前端类比 |
| --- | --- | --- | --- |
| 进程 Process | 操作系统分配资源的基本单位，有独立内存空间 | 每个 `svc-*` 启动后是一个 JVM 进程 | 一个独立运行的 Node 程序 / 一个浏览器标签页进程 |
| 线程 Thread | 进程内的执行单元，**共享进程的内存** | Tomcat 池里每个处理请求的 `http-nio-xxx` 线程 | Web Worker（但 Worker 不共享内存，见 14.6） |

一句话记住：**进程之间隔离（各有各的内存），同一进程内的线程共享内存。**「共享内存」这四个字，就是 14.3 翻车的根本原因——多个线程能摸到同一个 `Account` 对象的 `balance`。

```text
   操作系统
   ┌────────────────────────┐   ┌────────────────────────┐
   │   进程 A (svc-user)     │   │   进程 B (svc-ai)       │
   │  ┌──────────────────┐  │   │  ┌──────────────────┐  │
   │  │   共享的堆内存    │  │   │  │   共享的堆内存    │  │
   │  │  (对象都在这里)   │  │   │  │                  │  │
   │  └────▲────▲────▲───┘  │   │  └──────────────────┘  │
   │   线程1 线程2 线程3     │   │   线程1  线程2          │
   │   (都能访问同一个对象)  │   │                        │
   └────────────────────────┘   └────────────────────────┘
       ↑ 进程内线程共享内存          ↑ 两个进程互不可见对方内存
```

**线程的创建有成本。** 每个 Java 线程默认要占约 1MB 的栈内存，创建/销毁还要操作系统介入。所以后端**绝不会每来一个请求就 `new Thread()`**，而是用**线程池**复用固定数量的线程——这正是下一章的内容。你现在只要知道：**线程不是免费的，不能无限开。**

> 💡 **前端类比**：这就像你不会为每个 API 请求都 `new` 一个全新的 axios 实例，而是复用同一个配置好的实例。线程池就是「线程界的连接池 / 实例池」，思路和 [第十三章 连接池](/back-end/frontend-backend-guide/13-connection-pools) 一脉相承。

## 14.5 最小代码：Thread 与 Runnable

后端日常很少手写 `Thread`（都交给框架的线程池了），但你得看得懂它，才能读懂别人的代码和日志。两种最基本的写法：

```java
// 写法一：实现 Runnable（推荐——只描述"要干的活"，和"怎么跑"解耦）
Runnable task = () -> {
    String name = Thread.currentThread().getName();
    System.out.println(name + " 正在生成图片...");
};

Thread worker = new Thread(task, "ai-worker-1");
worker.start(); // 注意：是 start() 不是 run()！

// 写法二：直接继承 Thread（少用，因为 Java 单继承很浪费）
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("我是一个独立线程");
    }
}
new MyThread().start();
```

**一个最容易踩的坑**：调 `start()` 才会**真正开新线程**；如果你误写成 `worker.run()`，那只是在**当前线程里普通地调一个方法**，根本没有并发——这种 bug 在日志里看「线程名全是 main」就能发现。

> 💡 **前端类比**：`Runnable` 就是你传给 `setTimeout(fn)` 的那个 `fn`——一段「待执行的逻辑」。区别在于：`setTimeout` 把 `fn` 排进**同一个主线程**的队列里晚点跑；`new Thread(fn).start()` 是真的**另起一个线程**和你并行跑。`run()` vs `start()` 的区别，就像「直接调用 `fn()`」和「把 `fn` 交给调度器去异步执行」的区别。

**主线程 vs 工作线程**：`main` 方法所在的线程叫**主线程**，是程序的入口。你 `start()` 出来的叫**工作线程**。在 14.3 的例子里，`main` 线程必须用 `t1.join()` 等两个工作线程都跑完，才能去读最终余额——否则可能在 worker 还没干完时就把结果打印了。

> 💡 **前端类比**：`t1.join(); t2.join();` 约等于 `await Promise.all([p1, p2])`——都是「等这几件事都完成了我再往下走」。

## 14.6 终极对比：JS「并发」 vs Java「并行」

这张表是本章最该带走的东西：

| 维度 | JS / 浏览器 / Node | Java / 后端 |
| --- | --- | --- |
| 执行模型 | 单线程 + 事件循环 | 多线程，线程池 |
| 「同时」的真相 | 并发（concurrency）：快速切换制造的错觉 | 并行（parallelism）：多核上真的同时跑 |
| 同一时刻跑几段代码 | 永远只有 1 段 | 可能几十上百段 |
| 共享变量被同时改写 | 不可能，天然安全 | **会，必须自己保证安全** |
| 需要加锁吗 | 几乎不用 | 经常需要 |
| 处理高并发靠什么 | 异步 I/O，单线程也能扛很多连接 | 多线程并行 + 线程池 |
| 调试难度 | 结果可复现 | 竞态导致结果**时对时错**，难复现 |

那 JS 里有没有「最接近多线程」的东西？有——**Web Worker**（以及 Node 的 `worker_threads`）。但它和 Java 线程有一个**决定性的不同**：

```text
   Web Worker（JS）                      Java 线程
   ┌───────────┐  ┌───────────┐         ┌───────────────────────┐
   │ 主线程     │  │ Worker     │         │      一个 JVM 进程      │
   │ 内存A     │  │ 内存B      │         │     共享的堆内存        │
   └─────┬─────┘  └─────┬─────┘         │   ┌─────────────────┐  │
         │              │               │   │  Account 对象    │  │
         └── postMessage─┘               │   └──▲────────▲─────┘  │
            （拷贝数据传递）              │   线程1      线程2     │
         各有各的内存，互不可见          │  （都直接读写同一对象）│
                                         └───────────────────────┘
```

- **Web Worker 不共享内存**：主线程和 Worker 之间只能用 `postMessage` **拷贝**数据来通信。正因为不共享，所以你也**不会**遇到 14.3 那种竞态——它在设计上就回避了这个问题。
- **Java 线程共享堆内存**：多个线程能直接拿到**同一个对象的引用**，同时读写它的字段。**共享是性能的来源（不用拷贝），也是危险的来源（会竞态）。**

> 💡 **一句话收束**：JS 用「不共享内存」换来了「不用操心线程安全」的清净；Java 用「共享内存」换来了高性能，代价是你必须亲自处理并发安全。前端的安全是**语言送你的**，后端的安全是**你自己挣的**。

## 14.7 动手验证（可选但强烈建议）

光看不练记不牢。如果你本地装了 JDK 17，可以亲手让它翻一次车。

**目标**：复现「丢失更新」，亲眼看到结果不等于 200000。

```bash
# 1. 把 14.3 的 Account 和 RaceDemo 两个类放进一个文件 RaceDemo.java
#    （把 public class Account 改成 class Account，去掉 public 即可同文件）
# 2. 直接用单文件模式运行（JDK 11+ 支持，无需先 javac）
java RaceDemo.java
```

**预期输出（多跑几次）：**

```text
期望值: 200000, 实际值: 191204
期望值: 200000, 实际值: 200000
期望值: 200000, 实际值: 187655
```

**怎么读**：只要出现**任何一次不等于 200000**，就证明了竞态真实存在。如果你电脑太快、循环次数太少导致每次都「蒙对」，把循环次数从 `100_000` 调大到 `1_000_000` 再试，翻车概率会显著上升。

**结论**：这段代码在前端（单线程 JS）里 100% 安全，在 Java 多线程里却会出错——**不是 Java 的 bug，是并发的本质**。记住这个手感，第十六章我们再用 `synchronized` / `AtomicInteger` / Redis 分布式锁把它修好。

## 小结

- **JS 是单线程 + 事件循环**：任意时刻只有一段代码在跑，所以前端天然不用担心「两段代码同时改一个变量」。它的「并发」是切换出来的错觉。
- **Java 后端是多线程并行**：一个 JVM 进程里 Tomcat 用线程池，每个 HTTP 请求分一个线程，成百上千个线程**同时执行同一份代码**。
- **共享内存是双刃剑**：同一进程内的线程共享堆内存，能直接读写同一个对象——这是高性能的来源，也是**竞态条件 / 线程安全问题**的根源，是前端没有的全新维度。
- **进程隔离、线程共享**；线程创建有成本（约 1MB 栈），所以后端用线程池复用而非每次 `new Thread`。
- **`start()` 才开新线程，`run()` 只是普通调用**；`join()` ≈ `await Promise.all`。Web Worker 虽是 JS 里最像线程的东西，但**不共享内存**，因此也避开了竞态——这恰恰反衬出 Java 共享内存的危险。
