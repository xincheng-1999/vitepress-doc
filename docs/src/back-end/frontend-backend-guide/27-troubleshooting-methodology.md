# 问题排查方法论

> 这一章讲的是排查问题的"心法和流程"，是后面[第 28 章 故障排查实战手册](/back-end/frontend-backend-guide/28-troubleshooting-playbook)的总纲。
> 手册告诉你"症状 X 该怎么治"，本章告诉你"拿到一个不认识的症状，该按什么顺序去想、去查"。
> 工具的具体用法（jstack、jmap、arthas、Prometheus 查询等）放在[第 29 章 诊断工具箱](/back-end/frontend-backend-guide/29-diagnostic-toolbox)。

线上出问题时，最值钱的能力不是"会用某个命令"，而是**遇到没见过的问题，能不慌、按一套固定流程逼近根因**。
前端类比：线上白屏报错时，老手不会瞎改代码，而是先打开 DevTools Network/Console，看是哪个请求挂了、哪行 JS 报错、是不是刚发的版引入的——后端排查也是同一套思维，只是工具从浏览器换成了日志、监控和服务器命令。

---

## 27.1 先记住五条心法

这五条不是口号，是无数次踩坑换来的肌肉记忆。每条都给一个反例，让你知道违反它会怎样。

| 心法 | 含义 | 违反后的典型翻车 |
| --- | --- | --- |
| **先看现象，别瞎猜** | 先把"谁挂了、挂成什么样"搞清楚 | 以为是 DB 慢，查了半天，其实是网关线程池满了 |
| **先看日志和监控，再动手** | 证据先行，假设其次 | 没看日志直接改代码重发，问题没解决还引入新 bug |
| **能复现就好办** | 优先找到稳定复现路径 | 偶发问题不复现就硬改，上线后照样炸 |
| **一次只改一个变量** | 改完观察，确认因果 | 同时改了配置 + 代码 + 重启，好了也不知道是哪个起的作用 |
| **改之前先想"最近发生了什么"** | 80% 的线上故障由最近的变更触发 | 盯着十年没动的老代码看，忽略了两小时前刚发的版 |

> 💡 **前端类比**：第五条对应你排前端 bug 时第一反应——"我刚才改了什么？"`git log`、`git diff` 一拉，问题往往就在最近三个 commit 里。后端的"最近变更"范围更大：发版、改配置、改数据库数据、扩缩容、依赖的下游服务发版，都算。

把第五条展开成一个排查前必问的清单，每次开查前先在脑子里过一遍：

```text
排查前的"最近变更"三连问
┌──────────────────────────────────────────────────┐
│ 1. 最近发版了吗？哪个服务？几点发的？             │
│    → 故障时间点和发版时间点对得上吗？             │
│ 2. 最近改配置/改数据了吗？                        │
│    → 限流阈值、连接池大小、开关、灰度名单、SQL... │
│ 3. 依赖是否异常？                                 │
│    → 下游服务、MySQL/MongoDB/Redis/RocketMQ、OSS、 │
│      外部 AI API 是不是也在同一时间抖动？         │
└──────────────────────────────────────────────────┘
```

## 27.2 标准排查流程

把排查拆成七步，遇事按这个顺序走，不要跳步。下面是整体决策流程图：

```text
┌─────────────────────────────────────────────────────────────┐
│ ① 确认现象与影响面                                          │
│    全挂还是部分挂？必现还是偶发？影响多少用户？             │
│    → 先决定要不要立刻止血（见 27.5）                        │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ ② 定位是哪个服务、哪一层                                    │
│    网关 / 应用 / DB / 下游服务 / 中间件 ？                  │
│    → 用 traceId 看链路耗时分布，二分法缩小范围              │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ ③ 收集证据：看日志 + 看监控                                 │
│    日志：error / 异常栈 / 关键业务日志                      │
│    监控：CPU / 内存 / RT / QPS / 错误率 / GC / 连接数       │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ ④ 提假设                                                    │
│    "我怀疑是 svc-ai 调外部 API 超时拖垮了线程池"            │
│    → 假设必须可验证，否则就是空想                           │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ ⑤ 验证假设（一次只改/查一个变量）                          │
│    验证成立 → 进入修复                                      │
│    验证失败 → 回到 ④ 换假设（别累加改动）                  │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ ⑥ 修复                                                      │
│    临时止血 + 根因修复，分清两者                            │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ ⑦ 复盘                                                      │
│    根因、影响、改进项、如何提前发现（加监控/告警）          │
└─────────────────────────────────────────────────────────────┘
```

> 💡 **前端类比**：这套流程就是 React 里调试一个渲染异常的放大版——先确认"是所有页面挂还是某个组件挂"（影响面），再用 React DevTools 定位是哪个组件（哪一层），看 console 报错（日志）和 Profiler（监控），猜是哪个 prop/state 引起的（假设），注释掉验证（验证），改好（修复），最后加个测试防回归（复盘）。

## 27.3 第①步：确认现象与影响面

动手前先把现象描述清楚。一句"系统挂了"没有任何排查价值，要拆成可量化的几个维度：

```text
现象描述四要素
  范围：全挂 / 部分服务挂 / 某个接口挂 / 某些用户挂
  频率：必现 / 偶发（多久一次）/ 突发后持续
  时间：从几点开始？和发版/流量高峰对得上吗？
  表现：报错码？超时？返回慢？数据不对？
```

举个 svc-canvas 的实例。运营反馈"提交生图任务一直转圈"。先别急着看 svc-canvas 代码，先确认影响面：

**症状**：部分用户提交生图任务后，前端轮询 `GET /api/canvas/v1/image/status/{taskId}` 长时间停在"处理中"。

**可复制的命令**（用网关日志按接口统计成功/失败比例，先量化"部分还是全部"）：

```bash
# 统计最近 1 万条 status 接口的 HTTP 状态码分布
grep "/api/canvas/v1/image/status" /data/logs/svc-gateway/access.log \
  | tail -10000 \
  | awk '{print $9}' \
  | sort | uniq -c | sort -rn
```

**预期输出样例**：

```text
   9210 200
    540 504
    250 200
```

**怎么读这段输出**：第一列是次数，第二列是 HTTP 状态码。绝大多数请求是 200（接口本身没挂），但有 540 次 504（网关等下游超时）。说明**不是全挂，是部分超时**——网关到 svc-canvas 这一跳有问题，而不是 svc-canvas 完全宕机。

**结论**：影响面是"部分请求超时"，必现性偏向"高峰期偶发"。这一步就把"系统挂了"收敛成了"网关访问 svc-canvas 部分超时"，排查范围一下小了一大圈。

> 确认影响面还决定了**要不要立刻止血**：如果是全站不可用，先按 27.5 止血再慢慢查；如果只是少量偶发超时，可以从容收集现场再修。

## 27.4 第②步：定位是哪个服务、哪一层

微服务最大的排查难点是"链路长"——一个请求穿过网关、好几个服务、多个中间件。前端只要看一个 Network 面板，后端要在一串服务里找出"慢/错"在哪一跳。三个核心技巧：

### 技巧一：用 traceId 看链路耗时分布

每个请求在网关入口会生成一个 `traceId`（参见[第 3 章 请求生命周期](/back-end/frontend-backend-guide/03-request-lifecycle)里讲的链路追踪），它会透传到每个服务的每条日志里。拿到一个慢请求的 traceId，把各服务这条链路的耗时拉出来，慢在哪层一目了然。

**目标**：定位 traceId 为 `a1b2c3d4` 的请求慢在哪一跳。

**可复制的命令**（去各服务日志里捞同一个 traceId，按时间排序）：

```bash
# 在日志聚合后的目录里，跨服务搜同一个 traceId
grep -rh "a1b2c3d4" /data/logs/svc-*/app.log | sort -k1,2
```

**预期输出样例**（精简过，只留时间戳、服务名和关键事件）：

```text
12:03:01.102 svc-gateway  收到 POST /api/canvas/v1/image/create traceId=a1b2c3d4
12:03:01.108 svc-canvas   ImageTaskServiceImpl.createImageTask start
12:03:01.110 svc-canvas   Feign 调用 svc-user 扣配额 start
12:03:06.090 svc-canvas   Feign 调用 svc-user 扣配额 end (cost=4980ms)
12:03:06.095 svc-canvas   发送 RocketMQ 任务消息 ok
12:03:06.099 svc-canvas   返回 RtData.ok(taskId=T_8821)
12:03:06.101 svc-gateway  响应 200 (total=4999ms)
```

**怎么读这段输出**：整条链路 4999ms。逐段算差值——网关到 svc-canvas 几毫秒，正常；`createImageTask start` 到"Feign 调 svc-user start"几毫秒，正常；**"Feign 调 svc-user 扣配额"耗时 4980ms**，几乎吃掉了全部耗时。

**结论**：慢点在 svc-canvas 调用 svc-user 这一跳，下一步直接去查 svc-user 的扣配额接口（很可能是它的数据库慢查询或锁等待），而不用再怀疑网关、RocketMQ、svc-ai。

> 💡 **前端类比**：这就是 DevTools Network 里的瀑布图（Waterfall）。一个接口慢，你会展开看是 Queueing、TTFB 还是 Content Download 慢。traceId 链路就是后端版的瀑布图，只不过"每个 span"是一个微服务调用。

### 技巧二：二分法缩小范围

不确定问题在链路前半段还是后半段时，从中间砍一刀。比如怀疑是 svc-canvas 自身慢还是它的下游慢：直接绕过下游，给 svc-canvas 一个 mock 配额响应或打桩，看接口还慢不慢。慢 → 问题在 svc-canvas 自身；不慢 → 问题在下游。每砍一刀排除一半可能。

### 技巧三：对比正常实例与异常实例

svc-ai 部署了 3 个 Pod，只有 1 个 Pod 的请求超时。把异常 Pod 和正常 Pod 放一起对比，差异点就是嫌疑点：

```text
对比维度          正常 Pod (ai-2)      异常 Pod (ai-1)
─────────────────────────────────────────────────────
CPU               35%                  98%        ← 差异
堆内存            1.2G / 2G            1.95G / 2G  ← 差异
线程数            220                  512(满)     ← 差异
最近 Full GC      2 小时前             每 30 秒一次 ← 差异
启动时间          昨天                 昨天        相同
镜像版本          v1.8.3               v1.8.3      相同
```

**怎么读**：镜像和启动时间都一样，排除"发了不同的版"。CPU 飙高 + 堆快满 + Full GC 频繁 + 线程满——这是一组互相印证的现象，典型的"内存吃紧导致 GC 风暴拖垮一切"（具体怎么治见[第 20 章 OOM 与内存泄漏](/back-end/frontend-backend-guide/20-oom-memory-leak)）。

**结论**：单实例内存问题，而非全局逻辑错误。可以先把这个 Pod 摘流量保现场，再慢慢分析它的堆。

## 27.5 止血 vs 根治：分清两件事

线上排查有一个绕不开的矛盾：**重启/扩容能最快恢复服务，但会丢现场**。处理原则是把"止血"和"留现场"这两件事分开做、按顺序做。

```text
故障升级判断
        │
   全站不可用 / 大面积影响 ?
        │
    ┌───┴────────────────────────┐
   是                            否
    │                            │
    ▼                            ▼
立刻止血保服务              从容留现场再查
(可接受丢现场)             (重启前先 dump)
    │                            │
但止血前若有余力，          ├ 留线程栈 dump（jstack）
尽量先抓一份现场：          ├ 留堆 dump（jmap，可能较慢）
├ 至少摘 1 个异常实例        ├ 复制相关时间段日志
│  出来留着，其余重启        └ 截图监控曲线
└ 抓一份 jstack 再重启        然后再决定改什么
```

几个反复强调的点：

- **重启大法治标不治本**：内存泄漏、连接泄漏、死锁，重启后过段时间还会复发。重启只是争取排查时间，不是结论。
- **止血时可以先重启，但尽量留一个"案发现场"**：3 个 Pod 都炸了，别一股脑全重启——摘 1 个出来（在 K8s 里可以删掉它的 label 让它脱离 Service，参见[第 24 章 Kubernetes 实战](/back-end/frontend-backend-guide/24-kubernetes-in-practice)），dump 完再处理。重启会清空 JVM 进程，线程栈和堆里的证据全没了。
- **改代码不改配置/数据是常见误区**：很多"代码 bug"其实是配置写错、限流阈值设小了、灰度名单漏了某用户、或者数据库里有一条脏数据。改之前先确认问题到底出在代码、配置还是数据。

下面是一段"重启前先留现场"的标准操作。

**目标**：异常 Pod `svc-ai-1` 要重启止血，重启前留一份线程栈和堆。

**可复制的命令**：

```bash
# 1) 找到 Java 进程 PID
jps -l
# 2) 抓线程栈（很快，几乎不影响进程）
jstack <pid> > /data/dump/svc-ai-1-jstack-$(date +%H%M%S).txt
# 3) 抓堆 dump（较慢，大堆可能卡几秒，会 stop-the-world）
jmap -dump:live,format=b,file=/data/dump/svc-ai-1-heap.hprof <pid>
# 4) 现场留好了，再重启容器
kubectl rollout restart deployment/svc-ai
```

**预期输出样例**（jstack 片段，能看出线程都卡在哪）：

```text
"http-nio-8080-exec-198" #198 daemon prio=5 ... WAITING on condition
   java.lang.Thread.State: WAITING (parking)
        at jdk.internal.misc.Unsafe.park(Native Method)
        - parking to wait for <0x000000076b2f1a30> (a ...FutureTask)
        at com.demo.svcai.client.ExternalAiApiClient.generate(ExternalAiApiClient.java:88)
```

**怎么读这段输出**：几十上百个 `http-nio-*-exec` 线程都 `WAITING`，且都卡在同一行 `ExternalAiApiClient.generate`——说明 tomcat 工作线程全被"调外部 AI API"堵住了，新请求没线程可用，于是超时。这就把假设直接坐实成"外部 AI API 慢/超时配置缺失，拖垮了线程池"（线程池相关见[第 15 章 线程池](/back-end/frontend-backend-guide/15-thread-pools-executor)）。

**结论**：根因是外部依赖超时 + 缺超时/熔断保护，重启只能临时恢复，必须补超时和熔断才算根治。现场已经留好，可以放心重启止血了。

## 27.6 提假设与验证：一次只改一个变量

收集完证据后进入"假设—验证"循环。这一步最容易翻车，记住两个纪律。

**纪律一：假设必须可验证。** "可能是网络问题"不是一个可验证的假设，"svc-canvas 到 svc-user 的网络 RT 超过 200ms"才是——后者能用 `ping` / `curl -w` 直接量。

**纪律二：一次只改一个变量。** 改完观察结果，再决定下一步。

下面是一个错误示范和正确示范的对比：

```text
错误做法（同时改三样）              正确做法（逐个验证）
─────────────────────────          ─────────────────────────
1. 调大连接池                       1. 只调大连接池 → 观察 → 没好
   + 同时改重试逻辑                 2. 回滚，只加超时配置 → 观察 → 好了
   + 同时重启所有实例               3. 确认根因是缺超时配置
→ 好了也不知道哪个起作用            → 因果清晰，能写进复盘
→ 没好不知道排除了什么              → 每一步都排除了一种可能
```

**验证假设的命令样例**（验证"svc-user 扣配额接口本身慢"这个假设，绕过链路直接压它）：

```bash
# 直接调 svc-user 的扣配额接口，看单次耗时
curl -s -o /dev/null -w "http_code=%{http_code} time_total=%{time_total}s\n" \
  -X POST http://svc-user:8080/internal/quota/deduct \
  -H "Content-Type: application/json" \
  -d '{"uid":12345,"amount":1}'
```

**预期输出样例**：

```text
http_code=200 time_total=4.812s
```

**怎么读**：HTTP 200 说明接口逻辑没报错，但 `time_total=4.812s` 说明它本身就慢。结合 27.4 拿到的链路数据（这一跳 4980ms），二者吻合——假设成立，慢确实在 svc-user 的扣配额接口，而不是网络或调用方。下一步顺着去查它的 SQL 慢查询或锁等待（见[第 10 章 SQL 与索引](/back-end/frontend-backend-guide/10-sql-and-indexes)、[第 11 章 事务与一致性](/back-end/frontend-backend-guide/11-transactions-consistency)）。

## 27.7 第⑦步：复盘

问题解决不等于排查结束。复盘是把这次的经验沉淀成团队资产，至少回答四个问题：

```text
复盘四问
  1. 根因是什么？（一句话能说清，比如"外部 AI API 无超时配置，高峰期拖垮线程池"）
  2. 影响范围多大？（多少用户、多长时间、有没有资损）
  3. 改进项有哪些？（补超时+熔断、连接池监控、压测覆盖）
  4. 怎么提前发现？（加什么监控/告警，让下次在爆炸前就告警）
```

第 4 点最重要：好的复盘不是"以后大家小心点"，而是落到具体的监控和告警上——比如给"tomcat 线程池使用率 > 80%"加告警（监控告警体系见[第 30 章 可观测性](/back-end/frontend-backend-guide/30-observability)）。这样同类问题下次会在变成故障前就被发现。

> 💡 **前端类比**：复盘的"加监控"就像你修完一个线上 bug 后补一个单元测试 + 在 Sentry 里加一条告警规则——不是为了这次，是为了下次能更早发现。

## 27.8 一个完整的排查 walkthrough

把前面的流程串起来，再走一遍开头那个"生图任务一直转圈"的案例，体会节奏：

```text
① 确认现象：部分 status 请求 504，高峰偶发（27.3 的 awk 统计）
   → 不是全挂，先不重启，从容查
② 定位层级：拿一个慢请求的 traceId 拉链路（27.4 技巧一）
   → 慢在 svc-canvas → svc-user 扣配额这一跳，4980ms
③ 收集证据：看 svc-user 日志和监控
   → 该时段扣配额接口 RT 飙升，DB 慢查询日志有命中
④ 提假设：svc-user 扣配额接口因 SQL 慢查询变慢
⑤ 验证：直接 curl 扣配额接口（27.6）→ 4.8s，假设成立
   → 进一步 explain 该 SQL，发现少了 uid 索引
⑥ 修复：先临时给该接口加缓存止血，再补索引根治（只改一处，观察）
⑦ 复盘：根因=缺索引；改进=慢查询告警+上线前 explain 检查
```

注意全程：**没在第一步就重启**（影响面可控）、**用 traceId 一刀定位**（没逐个服务乱翻）、**验证时只动一个变量**（先缓存后索引分开做）、**最后落到告警**（防复发）。这就是方法论的价值——把一次慌乱的"线上着火"变成一条可复述、可教学的处理路径。

---

## 小结

- 排查靠的是流程而非运气：**确认现象 → 定位层级 → 看日志和监控 → 提假设 → 验证 → 修复 → 复盘**，遇事按这七步走，不跳步。
- 五条心法刻进肌肉：先看现象别瞎猜、证据先行、能复现就好办、一次只改一个变量、先想"最近发生了什么"。
- 定位层级三板斧：用 **traceId 看链路耗时分布**（后端版瀑布图）、**二分法**砍范围、**对比正常实例与异常实例**找差异。
- 止血和根治要分开：全站不可用先重启/扩容保服务，但**重启会丢现场**，有余力就先 jstack/jmap dump 一份再重启；重启治标不治本，别拿它当结论。
- 常见误区：一上来就重启、只改代码不查配置和数据、忽略最近的变更（80% 的故障由最近变更触发）。

### 自测

1. 接到"系统挂了"的反馈，你动手查之前要先把现象拆成哪几个维度？为什么影响面会决定要不要立刻重启？
2. 一个微服务请求很慢，你怎么用 traceId 判断慢在哪一跳？这和前端 DevTools 的什么功能对应？
3. 为什么"一次只改一个变量"如此重要？同时改了连接池、重试和重启三样，问题好了，会带来什么后果？
